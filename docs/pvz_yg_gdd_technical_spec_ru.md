# PvZ YG

## Полный документ игрового дизайна (GDD) и техническая спецификация

**Платформа:** веб (Яндекс Игры)  
**Локализация:** только русский язык  
**Основа геймплея:** базовые механики Plants vs. Zombies

---

## ЧАСТЬ I. ДОКУМЕНТ ИГРОВОГО ДИЗАЙНА (GDD)

### 1. Обзор продукта

**PvZ YG** — соревновательная игра на выживание с асинхронными PvP-фазами.

Ключевой формат матча:

- 10 игроков в одной сессии
- до 100 волн
- прогрессия сложности по тирам
- периодические PvP-волны по конфигурации
- постоянная мета-прогрессия через алмазы
- глобальная и недельная таблицы лидеров

Дизайн и пользовательский опыт изначально учитывают требования модерации Яндекс Игр.

### 2. Основной игровой цикл

1. Игрок нажимает **«Начать игру»**.
2. Запускается матчмейкинг:
   - режим 1: случайный
   - режим 2: по MMR (рейтинг подбора соперников)
3. Лобби заполняется до 10 участников (боты при недоборе).
4. Стартует фаза выживания.
5. PvP-волны запускаются по таблице из ресурсов.
6. Матч продолжается до одного из условий:
   - пройдена 100-я волна
   - игрок выбывает

Если в матче остаётся только один живой игрок:

- матч продолжается
- PvP отключается
- игрок фармит очки и алмазы
- место в лидерборде продолжает расти

### 3. Механики режима выживания

#### 3.1 Случайные семена

- доступные растения рероллятся каждые `N` секунд
- фиксированной колоды до матча нет
- акцент на адаптивную стратегию

#### 3.2 Удобрения (перманентные апгрейды в рамках матча)

Типы бонусов:

- усиление урона
- ускорение атаки
- прирост HP
- буст генерации солнца
- расширение радиуса действия

Шанс выпадения и сила апгрейда масштабируются по тирам.

### 4. PvP-система

#### 4.1 Триггеры PvP

PvP-волны запускаются по внешней конфигурации в игровых ресурсах.

#### 4.2 Поведение во время PvP-фазы

- камера отдаляется
- игрок видит поле случайного соперника
- защищающийся игрок продолжает стандартный геймплей
- наблюдающий игрок не может сажать растения
- наблюдающий игрок может спавнить дополнительных врагов

#### 4.3 Риск/награда

- чем больше дополнительных врагов отправлено,
- тем выше потенциальная награда по солнцу для защитника при успешной обороне

#### 4.4 Смена ролей

На следующей PvP-волне пары меняются ролями:

- бывший наблюдатель становится защитником
- бывший защитник становится наблюдателем

Это снижает влияние случайности и повышает честность результата.

### 5. Тировая прогрессия

Матч делится на тиры. Каждый тир включает:

- рост базовой сложности врагов
- подключение новых типов зомби
- завершающую босс-волну

Боссы:

- имеют уникальные способности
- обладают повышенным запасом здоровья
- требуют адаптации билда и таймингов

### 6. Правила завершения матча

- Если несколько игроков пережили 100-ю волну, побеждает игрок с максимальным счётом.
- Если один игрок остаётся раньше 100-й волны, он продолжает игру в соло-режиме:
  - получает дополнительные очки рейтинга
  - получает бонусные алмазы

### 7. Мета-прогрессия

#### 7.1 Валюта: алмазы

Источники:

- итоговое место в матче
- глубина выживания
- убийства боссов
- соло-фаза после выбывания остальных игроков

#### 7.2 Межматчевые улучшения

За алмазы открываются:

- стартовые модификаторы характеристик
- повышение генерации солнца
- небольшие множители урона
- дополнительные рероллы вариантов растений

Прогресс хранится на сервере.

### 8. Монетизация (только через Yandex SDK)

Покупки:

- наборы алмазов
- эксклюзивная косметика:
  - скины растений
  - скины зомби
  - темы поля
  - темы интерфейса

Механики «плати и побеждай» за пределами допустимых стат-усилений запрещены.  
Все платежи только через SDK Яндекс Игр.

### 9. Лидерборды

Типы:

- глобальный
- недельный (с авто-сбросом)

Метрика ранжирования:

- общий счёт
- глубина выживания
- модификатор PvP-эффективности

Интеграция выполняется через API лидербордов Yandex SDK.

### 10. Пользовательский опыт и соответствие требованиям Яндекс Игр

Обязательные требования:

- звук автоматически ставится на паузу при потере фокуса вкладки
- отсутствие обязательной внешней регистрации
- стабильная работа на десктопных и мобильных браузерах
- понятный обучающий блок по управлению
- средняя длительность сессии более 10 минут
- русский язык интерфейса и текстов
- отсутствие сторонних платёжных шлюзов

## ЧАСТЬ II. ТЕХНИЧЕСКАЯ СПЕЦИФИКАЦИЯ

### 1. Архитектура

**Клиент (Web/HTML5/WebGL):**

- рендеринг
- ввод игрока
- клиентская визуализация состояний
- отправка команд на сервер

**Сервер (источник истины):**

- матчмейкинг
- расчёт и валидация ключевой логики
- синхронизация PvP-фаз
- расчёт MMR
- хранение и обновление алмазов
- античит-проверки

**База данных:**

- профиль игрока
- рейтинг MMR
- баланс алмазов
- владение косметикой
- недельные срезы результатов

### 2. Система матчмейкинга

#### 2.1 Случайный режим

- подбор из общей очереди
- быстрый старт за счёт ботов при нехватке людей

#### 2.2 Режим MMR (рейтинг подбора соперников)

- ELO-подобный рейтинг
- окно допуска по рейтингу: `MMR ± Δ`
- параметр `Δ` настраивается на сервере
- обновление MMR после завершения матча

### 3. Обязанности сервера

Сервер должен валидировать:

- количество отправляемых врагов в PvP
- генерацию ресурсов (солнца)
- множители от апгрейдов
- легитимность боевых расчётов

Сервер также обрабатывает:

- дисконнект игрока: мгновенное выбывание (без реконнекта)
- точную синхронизацию начала/конца PvP-фаз

### 4. Античит

- серверная генерация солнца (источник истины — сервер)
- серверный расчёт урона (источник истины — сервер)
- серверные лимиты спавна врагов
- журнал подозрительных действий

Рекомендуемые события античита:

- резкие аномалии прироста ресурсов
- превышение лимитов действий в интервале
- рассинхрон клиент/сервер по критическим параметрам

### 5. Упрощённая модель данных

**Сущность «Игрок»:**

- `id`
- `diamonds`
- `mmr`
- `cosmeticsOwned`
- `weeklyScore`

**Сущность «Матч»:**

- `lobbyId`
- `players`
- `currentWave`
- `tier`
- `pvpScheduleIndex`

### 6. Целевые показатели производительности

- загрузка стартового бандла: менее 3 секунд (целевое значение)
- размер сборки: желательно менее 150 МБ
- производительность: 60 FPS на десктопе
- минимально допустимо: 30 FPS на мобильных устройствах

### 7. Точки интеграции Yandex SDK

- инициализация при загрузке
- отправка результатов в лидерборд
- обработка платежей
- вознаграждаемые рекламные форматы (опционально, за алмазы)
- авторизация игрока (опционально через Yandex ID)

### 8. Масштабируемость

- инстансы матчей без сохранённого локального состояния
- горизонтальное масштабирование игровых серверов
- Redis для оперативного состояния сессий
- отдельный сервис лидерборда

### 9. Будущие расширения

- ранговые сезоны
- новые тиры растений
- событийные модификаторы
- кооперативный режим

**PvZ YG** — соревновательный сессионный проект с акцентом на:
